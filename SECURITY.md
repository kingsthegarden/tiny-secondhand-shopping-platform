# 개발 과정에서 확인한 보안 약점과 조치 내역

개발 단계별로 "기능만 구현했을 때 생기는 보안 약점"을 식별하고, 어떻게 제거했는지 기록한 문서입니다.
(보고서의 "확인한 보안 약점 및 변경 내용" 섹션의 근거 자료)

## 1. 인증(회원가입/로그인)

| # | 보안 약점 (그대로 두면) | 조치 내용 | 코드 위치 |
|---|---|---|---|
| 1-1 | 비밀번호 평문 저장 → DB 유출 시 전 계정 탈취 | Werkzeug `generate_password_hash`(scrypt, 고유 salt)로 해시 저장. 어디에도 평문 저장·로깅 없음 | `routes/auth.py`, `routes/users.py` |
| 1-2 | 아이디/비밀번호 형식 미검증 → 특수문자 아이디, 1자리 비밀번호 허용 | 아이디 `[A-Za-z0-9_]{3,20}` 정규식, 비밀번호 8~64자 + 영문·숫자 필수 (서버측 WTForms 검증) | `forms.py` |
| 1-3 | 로그인 무차별 대입(brute force) 가능 | 계정별 5회 실패 시 5분 잠금 + IP별 분당 10회 제한 | `routes/auth.py`, `utils.py` |
| 1-4 | 로그인 실패 사유 구분 노출 → 아이디 존재 여부 열거(enumeration) | 아이디 오류/비밀번호 오류 모두 동일한 메시지 반환 | `routes/auth.py` |
| 1-5 | 세션 고정(Session Fixation) 공격 | 로그인 성공 시 `session.clear()` 후 새 세션 발급 | `routes/auth.py` |
| 1-6 | `next` 파라미터 오픈 리다이렉트 → 피싱 사이트로 이동 | 상대 경로(`/`로 시작, `//` 제외)만 허용 | `routes/auth.py` |
| 1-7 | 하드코딩된 SECRET_KEY → 세션 위조 가능 | 환경변수 또는 최초 실행 시 랜덤 생성해 `instance/`(git 제외)에 0600 권한으로 저장 | `config.py` |
| 1-8 | 로그인 응답 **시간차**로 계정 존재 여부 열거 가능 → 존재하지 않는 아이디는 비밀번호 해시 계산을 건너뛰어 응답이 더 빨랐음(에러 메시지는 동일해도 타이밍으로 구분 가능) | 존재하지 않는 사용자에 대해서도 고정 더미 해시로 `check_password_hash`를 항상 실행해 응답 시간을 동일하게 맞춤 | `routes/auth.py` (`_DUMMY_PASSWORD_HASH`) |
| 1-9 | 회원가입에 레이트리밋이 없어 자동화된 대량 계정 생성(스팸/자원 고갈) 가능 | IP별 시간당 5회로 제한 | `routes/auth.py` |
| 1-10 | 길이·형식 조건만 통과하면 `password1`, `12345678` 같은 초흔한 비밀번호도 허용 | 공개 유출 비밀번호 빈도 상위권 패턴 기반 차단목록 추가 검증 | `forms.py` (`_COMMON_WEAK_PASSWORDS`) |

## 2. 세션/쿠키

| # | 보안 약점 | 조치 내용 | 코드 위치 |
|---|---|---|---|
| 2-1 | 세션 쿠키를 JS로 읽을 수 있음 → XSS 시 세션 탈취 | `HttpOnly` 플래그 적용, HTTPS 환경용 `Secure` 옵션 제공 | `config.py` |
| 2-2 | 세션 무제한 유지 → 방치된 세션 악용 | 30분 유휴 시간 만료(`PERMANENT_SESSION_LIFETIME`) | `config.py` |
| 2-3 | 비밀번호 변경 시 재인증 없음 → 세션 탈취자가 계정 완전 탈취 | 현재 비밀번호 재확인 후에만 변경 허용 | `routes/users.py` |
| 2-4 | 휴면/차단 계정의 기존 세션이 계속 유효 | 매 요청마다 계정 상태 확인, 비활성이면 즉시 세션 파기 | `market/__init__.py` |

## 3. 입력값 처리 (인젝션/XSS)

| # | 보안 약점 | 조치 내용 | 코드 위치 |
|---|---|---|---|
| 3-1 | SQL 인젝션 (검색어, 로그인 폼 등) | 모든 쿼리를 SQLAlchemy ORM + 바인딩 파라미터로 작성. 문자열 조합 쿼리 없음 | 전체 |
| 3-2 | 검색어의 `%`, `_` 와일드카드 남용 → 의도치 않은 전체 매칭 | LIKE 이스케이프 처리 후 바인딩 | `routes/main.py`, `routes/admin.py` |
| 3-3 | 상품 설명·소개글·채팅에 스크립트 삽입(Stored XSS) | Jinja2 autoescape 유지(비활성화 코드 없음), `\|safe` 미사용. 채팅은 클라이언트에서도 `textContent`로만 렌더링 | 템플릿 전체, `static/js/chat.js` |
| 3-4 | 폼 길이/형식 미검증 → 초대형 입력, 음수 가격 등 | 모든 폼 필드에 서버측 길이·범위·형식 검증 (가격 0~1억, 금액 1~1천만 등) | `forms.py` |
| 3-5 | XSS가 성공해도 피해 최소화 필요 (심층 방어) | CSP `script-src 'self'` (인라인 스크립트 전면 차단, 모든 JS 외부 파일화) | `market/__init__.py` |

## 4. CSRF

| # | 보안 약점 | 조치 내용 | 코드 위치 |
|---|---|---|---|
| 4-1 | 모든 폼이 CSRF에 취약 (송금·비밀번호 변경 포함) | Flask-WTF `CSRFProtect` 전역 적용, 모든 POST에 토큰 필수. 로그아웃도 POST로 구현 | `market/__init__.py`, 템플릿 전체 |
| 4-2 | 쿠키만으로 요청 신뢰 | `SameSite=Lax` 쿠키 + CSRF 토큰 이중 방어 | `config.py` |

## 5. 접근 제어 (인가)

| # | 보안 약점 | 조치 내용 | 코드 위치 |
|---|---|---|---|
| 5-1 | URL만 알면 남의 상품 수정/삭제 가능 (IDOR) | 소유자 또는 관리자인지 서버측 확인, 아니면 403 | `routes/products.py` |
| 5-2 | 일반 사용자가 `/admin` 접근 | `admin_required` 데코레이터로 역할 검사 | `utils.py`, `routes/admin.py` |
| 5-3 | 미로그인 사용자가 기능 접근 | `login_required` 데코레이터 일괄 적용 | `utils.py` |
| 5-4 | 타인의 1:1 채팅방 참여/도청 | Socket 연결 시 인증 확인 + join/send 시 방 멤버십 서버측 검증 | `sockets.py`, `utils.py` |
| 5-5 | 관리자가 사용자 관리 화면에서 **자기 자신의 role/status도 변경 가능** → 실수로 자기 권한을 `user`로 낮추면 그 관리자가 유일한 관리자였을 경우 시스템에 관리자가 0명이 되어 앱 안에서 복구 불가(서버 CLI 재실행 필요). 기존에는 status만 자기 자신 변경을 막고 있었고 role은 막혀 있지 않았음 | 자기 자신의 계정은 이 화면에서 role/status 변경 자체를 완전히 차단(다른 관리자 계정 사용 유도). CSRF로 보호되어 외부 공격 벡터는 아니지만, 실수로 인한 자기 잠금(self-lockout) 가용성 문제였음 | `routes/admin.py` |

## 남아있는 한계 (운영 시 추가 필요)

- 개발 서버(Werkzeug) 사용 중 — 운영 시 gunicorn + eventlet/gevent 및 HTTPS(리버스 프록시) 필요
- 레이트리밋이 프로세스 메모리 기반 — 다중 프로세스 운영 시 Redis 등 외부 저장소 필요
- SQLite 단일 파일 DB — 동시성 요구가 커지면 PostgreSQL 등으로 이전, DB 계정 최소 권한 설정 필요
- 이메일 인증·2FA 미구현 (과제 범위 외)
